# Plan: Horarios — HU-008

> Fecha: 2026-05-25
> HU: HU-008 (Generacion automatica de horarios)
> Modulo destino: `src/modules/horarios/`

## Resumen

Modulo completo de generacion automatica de horarios usando algoritmo CSP de 9 fases. Incluye: entidades de dominio (Horario, Asignacion), motor del algoritmo con scoring docente y validacion de restricciones duras/blandas, validacion pre-ejecucion, progreso en tiempo real, grilla visual del resultado, y resumen estadistico. Solo el Director puede ejecutar, solo en estado "Generacion" del periodo.

## Archivos a crear

### domain/entities/
- `horario.entity.ts` — Interface `Horario` (id, periodoId, estado: 'borrador'|'aprobado'|'publicado', fechaGeneracion, resumen stats) y `Asignacion` (id, horarioId, grupoId, docenteId, aulaId, dia, bloque, tipo: 'teorico'|'practico')
- `scoring.entity.ts` — Funciones puras de calculo de puntaje docente: `calculateDocenteScore()`, `getCategoriaValue()`, `getAntiguedadValue()`, `getDisponibilidadValue()`, `getPreferenciaValue()`, `getCargaValue()`

### domain/services/
- `constraint-validator.service.ts` — Validador de restricciones duras: no simultaneidad docente, no simultaneidad aula, no superposicion ciclo, capacidad aula, tipo aula compatible. Recibe estado parcial del horario y devuelve `{valid, violations[]}`
- `schedule-generator.service.ts` — Motor CSP de 9 fases. Funcion principal `generateSchedule()`. Recibe todos los datos (docentes, cursos, grupos, aulas, disponibilidad, restricciones aula) y devuelve `{asignaciones[], resumen, unassigned[]}`. Emite callbacks de progreso por fase

### domain/repositories/
- `horario.repository.ts` — Interface `IHorarioRepository` (save, findByPeriodo, findById, saveAsignaciones, deleteAsignacionesByHorario)

### application/dtos/
- `generate-horario.dto.ts` — Schema Zod: periodoId (uuid), opcional `forceWithoutFullAvailability` (boolean, default false, para RN-024)

### application/use-cases/
- `validate-pre-generation.use-case.ts` — Fase 1 del algoritmo: verifica periodo en "Generacion", todos los docentes con disponibilidad, todos los cursos completos, aulas disponibles. Retorna `{valid, errors[]}`
- `generate-horario.use-case.ts` — Orquesta el algoritmo completo: llama validate, luego schedule-generator, guarda resultado en BD. Retorna Horario + Asignaciones + resumen
- `get-horario.use-case.ts` — Obtener horario con asignaciones por periodo

### infrastructure/
- `supabase-horario.repository.ts` — Implementa IHorarioRepository. Lee/escribe tablas `horarios` y `asignaciones`

### presentation/actions/
- `validate-generation.action.ts` — Server Action: valida pre-condiciones. Auth: director
- `generate-horario.action.ts` — Server Action: ejecuta generacion completa. Auth: director. Retorna horario + asignaciones + resumen
- `get-horario.action.ts` — Server Action: obtiene horario existente por periodo. Auth: director

### presentation/components/
- `generation-pre-check.tsx` — Panel de validacion pre-ejecucion: lista de checks (docentes, cursos, aulas) con estado verde/rojo por item
- `generation-progress.tsx` — Indicador de progreso por fase (1-9) con nombre de fase actual y barra
- `horario-grid.tsx` — Grilla visual del horario generado: tabla DIAS x BLOQUES, celdas con curso+docente+aula, filtrable por ciclo
- `generation-summary.tsx` — Resumen estadistico: cursos asignados/total, preferencias respetadas %, distribucion carga, unidades sin asignar
- `horarios-content.tsx` — Contenedor principal: orquesta pre-check → boton generar → progreso → grid + resumen. 4 estados

### barrel
- `src/modules/horarios/index.ts` — Exports publicos

## Riesgos / dudas

1. **Tablas BD**: la HU dice que `horarios` y `asignaciones` existen. No tenemos schema exacto de columnas. Asumimos: `horarios` (id, periodo_id, estado, fecha_generacion, resumen, created_at, updated_at) y `asignaciones` (id, horario_id, grupo_id, docente_id, aula_id, dia, bloque, tipo, created_at). Si difieren, ajustar en infraestructura.
2. **Progreso en tiempo real (RF-033)**: el algoritmo corre en una Server Action (server-side). Para progreso real se necesitaria SSE o polling. Como primera version, el componente puede mostrar progreso simulado por fases (la accion retorna el resultado completo). Si el usuario quiere progreso real via SSE, es una mejora posterior.
3. **Rendimiento (RNF-001)**: el algoritmo debe correr en <=30s para 50 docentes, 60 cursos, 20 aulas. El domain service es logica pura en memoria, deberia ser rapido. Si es lento, se puede optimizar despues.
4. **RN-024**: si faltan docentes sin disponibilidad, el Director puede forzar la generacion con el flag `forceWithoutFullAvailability`. Los docentes sin disponibilidad se excluyen.
5. **Afinidad docente-curso (RN-031)**: no hay tabla de afinidad en el sistema actual. Como primera version, se omite esta restriccion blanda. Se puede agregar despues.
6. **Pagina app router**: igual que disponibilidad, no hay `src/app/` visible. El usuario debe crear la pagina.
